# Data Dictionary — Synthetic Datasets

Generated by `src/ingestion/generate_synthetic_data.py` (seed=42, deterministic).
Regenerate anytime with:
```
python3 src/ingestion/generate_synthetic_data.py
```

## market_timeseries.csv
Daily market/macro series, ~3 years, with a regime-switching model (bull/bear/volatile).

| column | type | notes |
|---|---|---|
| date | date | daily |
| regime | str | bull / bear / volatile — Markov chain, persistent regimes |
| equity_index | float | starts at 100, regime-dependent drift & vol |
| bond_index | float | starts at 100, low vol |
| volatility_index | float | proxy for VIX-like measure |
| interest_rate | float | random-walk, clipped [-2, 5] |
| inflation_rate | float | random-walk, clipped [-2, 6] |

## recommendations_log.csv — **the core causal dataset**
One recommendation event per user. This is what the uplift model and CF
baseline both train/score against.

| column | type | notes |
|---|---|---|
| user_id | int | FK -> users.csv |
| rec_date | date | date the recommendation was made |
| market_regime_at_rec | str | regime at time of recommendation |
| **treatment** | 0/1 | 1 = recommended increasing equity exposure, 0 = maintain/de-risk |
| **propensity** | float [0.05, 0.95] | P(treatment=1 \| X) under the *simulated existing advisor policy* — legitimately used by the offline IPS evaluator, but must NEVER be a model input feature (leakage) |
| **outcome_30d_return_pct** | float | realized 30-day forward return; the target variable |
| **true_ite** | float | ground-truth individual treatment effect — **validation only, never a feature or training target** |

### Why the causal design works
Treatment assignment is confounded by design: high risk-tolerance users in
bull markets are far more likely to receive treatment=1. A naive
treated-vs-control comparison on this data is biased (~+1.35pp in our
generated run) relative to the true average treatment effect (~+0.08pp).
This gap is the whole reason the project needs uplift/double-ML instead of
a simple comparison — if you regenerate data and this gap disappears,
something in the propensity model has been broken.

## Ground-rules for downstream teams (Days 2–4)
- **Never** use `propensity` or `true_ite` as a model input feature.
- `propensity` is fair game for the Day-4 offline IPS evaluator (that's
  literally what IPS needs).
- `true_ite` should only be used to sanity-check / validate model outputs
  offline, e.g., "does our estimated uplift correlate with the ground
  truth uplift" — never expose this in the served recommendation or dashboard.
